The Big Picture: What Is an Integration?

You already know Salesforce inside-out as an admin. An integration is simply two systems exchanging data automatically — no human clicking buttons in between.

🍽️ Analogy: the restaurant An API is like a waiter. You (the calling system) don't walk into the kitchen (the database) and grab food yourself. You give the waiter a structured order ("one SOQL query, please"), the waiter takes it to the kitchen, and brings back exactly what you asked for — nothing more. The menu is the API documentation: it tells you what you may order and how to phrase it.

Every integration answers three questions

  1. Who calls whom? An external system calls Salesforce (inbound), or Salesforce calls an external system (outbound). You'll do both.
  2. How does the caller prove who it is? That's authentication — the heart of this curriculum. OAuth 2.0 is the industry rulebook, and its "flows" are recipes for different situations.
  3. What data moves, in what shape? Almost always JSON over HTTP — both decoded in the next topic.
💡 You've been using APIs all along Data Loader, the Salesforce mobile app, Workbench, and every AppExchange package talk to your org through the same REST APIs you'll call by hand here. The only difference: now you make the raw call — the best way to see what's under the hood.

The universal ritual of the API economy

Every serious platform — Salesforce, Google, Dropbox, Auth0, Stripe — uses the same handshake:

  1. Register an app on the platform ("this external application exists").
  2. Receive a Client ID (the app's username) and usually a Client Secret (its password).
  3. Exchange those credentials for a temporary access token.
  4. Attach the token to every API call.

Learn this once and no "create an app to get credentials" page will ever intimidate you again. In Salesforce, step 1 is called an External Client App — it gets a full deep-dive in topic 5.

🧠 Check yourself: your marketing tool creates Leads in Salesforce every night at 2 AM. Is that inbound or outbound — and is a human involved?
Inbound (the external tool calls Salesforce) and no human is involved — which, as you'll see in topic 3, points straight at the client credentials flow.

client credentials salesforce		The Salesforce user an inbound client-credentials integration acts as. Same mental model as a dashboard's running user: the token gets exactly that user's permissions and sharing.

client credentials salesforce		A dedicated Salesforce user account used only by an integration, never a human. Gives clean audit trails and least-privilege control.
Web Server Flow (Authorization Code)
web server		The recipe for apps acting on behalf of a logged-in human: user logs in on the platform's own page, app receives a code, exchanges it for tokens. Module 3's star.
Authorization code
web server		A short-lived, single-use voucher handed back after login. Not a token — the app must exchange it (plus its own credentials) for real tokens.
Authorize endpoint
web server		The URL that shows the login + consent screen — a page a human visits, unlike the token endpoint that only machines call.
Consent screen
web server		The "This app wants to: … — Allow / Deny" page. What it lists = the scopes requested.
Callback / Redirect URI

Refresh token
web server		A long-lived credential issued alongside the access token (in this flow only). Trades for fresh access tokens silently — why your mobile apps stay logged in for months. Crown jewels.

web server		Optional hardening: every refresh issues a new refresh token and retires the old one.

web server		A random value sent into the login and checked on return — proves the response belongs to the request you started (anti-CSRF).

Claim (iss / sub / aud / exp)
jwt		One statement inside a JWT: who speaks (iss), acting as whom (sub), for which server (aud), until when (exp).
Public/private key pair
jwt		Two mathematically linked keys: what one signs, only the other verifies. Private key = signet ring (guard it); public certificate = a wax impression anyone may hold to check seals.
X.509 certificate
jwt		The standard file format wrapping a public key. What you upload to the platform that will verify your signatures.
assertion
jwt		OAuth's word for "a signed statement used instead of credentials" — the token-request parameter that carries the JWT.
Pre-authorization
jwt salesforce		Admin-side consent replacing the login screen: Permitted Users = "Admin approved users are pre-authorized" + access via profile/permission set. Mandatory for the JWT flow.
External Client App (ECA)
salesforce		Salesforce's current-generation app registration (Setup → External Client App Manager) for inbound access. Successor to Connected Apps. Full deep-dive in topic 5.
Connected App
salesforce		The older registration container. Existing ones still work, but creating new ones is restricted as of Spring '26 — use ECAs for new work.
My Domain
salesforce		Your org's unique URL (yourcompany.my.salesforce.com). The client credentials flow requires it — login.salesforce.com won't work for that token request.
Callout
salesforce		Salesforce lingo for an outbound API call — Apex or Flow calling an external system.
Named Credential
salesforce		Setup feature storing an external system's base URL — "the WHERE". Code just says callout:Name/path and Salesforce injects authentication automatically.
External Credential
salesforce		Setup feature storing how to authenticate (protocol, flow, token endpoint, credentials) — "the HOW". A Named Credential points at one.
Principal (on an External Credential)
salesforce		The named "identity slot" that actually holds the Client ID + Secret (encrypted). Users get access to it through a permission set — your admin lever over who may use the credentials.
External Auth Identity Provider
salesforce client credentials		A Setup record describing an outside authorization server (its token endpoint + any custom request parameters like Auth0's audience). External Credentials can reference it. You'll build one in Project B.
Remote Site Setting
salesforce		The older/simpler allowlist for outbound URLs. Needed for raw-URL callouts; not needed when you use a Named Credential.
Anonymous Apex
salesforce		Code run once from Developer Console (Debug → Open Execute Anonymous Window) without saving a class. Perfect for labs and smoke tests.
Queueable Apex
salesforce		Salesforce's way to run code asynchronously (in the background). Essential for us because triggers can't make callouts directly — the trigger enqueues a Queueable, which makes the callout moments later. Project C teaches it.
SOQL
salesforce		Salesforce Object Query Language — you know this one. In this curriculum you'll also run it through the REST API.

The Three Flows at a Glance

Three recipes, one rulebook. Click each tab to meet a flow — then the table below puts them side by side.

Machine to machine — no human anywhere

The app presents its Client ID + Secret to the token endpoint and gets an access token directly. Like a delivery driver with a standing contract: show the company badge, get a temporary key card.

  • Best for: middleware, ERP/marketing syncs, scheduled jobs — anything running unattended.
  • Identity: the app itself (in Salesforce, mapped to one fixed Run As user).
  • Taught in: Module 2 — with a full CRUD project.

A human logs in — the app acts as that person

The user logs in on the platform's own page and clicks Allow; the app receives a single-use code and exchanges it for tokens (including a refresh token). PKCE makes a stolen code worthless. The user's password never touches the app.

  • Best for: web/mobile apps, AppExchange apps, "Log in with…" buttons, anything touching "my data".
  • Identity: whoever logged in — their permissions, their audit trail.
  • Taught in: Module 3 (coming next).

Machine to machine — but with a signature, not a secret

The app writes a tiny signed message ("it's me, acting as this user, valid 3 minutes") using a private key. The platform verifies it with the registered certificate. No secret ever crosses the network.

  • Best for: CI/CD pipelines, ISV packages, enterprises that ban shared secrets.
  • Identity: any pre-authorized user named in the message.
  • Taught in: Module 4 (coming next).

The two tabs every app has

Patterns & Ground Rules

Five habits that separate professional integrations from time bombs. Every module applies all five.

1. Dedicated integration users (inbound)

Integrations get their own Salesforce user — never a human's account. Every record they touch shows that user in Created By; auditing and shutting down become one-click admin tasks; and nobody's departure kills a nightly sync. Minimal profile + one purpose-built permission set (API Enabled + only the object access needed).

2. The "WHERE vs HOW" split (outbound)

Outbound authentication lives in Setup, not in code:

  • Named Credential = the WHERE — the external system's base URL. Code says callout:Auth0_Mgmt_API/api/v2/users.
  • External Credential = the HOW — protocol, flow, token endpoint, and (via its Principal) the encrypted Client ID + Secret.

Result: no secret ever appears in Apex, credentials rotate without code changes, and a permission set controls which users' code may use them.

3. Verify in Postman before configuring Salesforce

Golden debugging habit: prove the third-party API works with raw requests first. Then, when the Salesforce side fails, you already know the credentials and endpoints are good — the bug is in your config, not their API. Every project in this curriculum follows this order.

4. Least privilege, everywhere

Fewest OAuth scopes that work · minimal integration-user permissions · one app per integration (so you can audit or kill just one) · one certificate per integration in Module 4.

5. Know your kill switches

Rotate the secret (Settings tab) · disable the flow or change Run As (Policies tab) · revoke a user's tokens (user detail → OAuth Connected Apps) · unassign the permission set that grants principal access. Offboarding an employee includes their OAuth grants.

🧠 Check yourself: a developer asks you to paste the Auth0 client secret into an Apex class "just for the demo". What's your answer, and where does it belong?
No — secrets never go in code (they end up in version control forever). It belongs in an External Credential principal, which encrypts it and gates access behind a permission set. The demo works the same via callout:.

Module 2 · Client Credentials

The Client Credentials Flow, Explained

One idea to hold on to: this flow is for machine-to-machine conversations. No login screen, no browser, no human. A server proves its identity with an ID + Secret and receives a temporary key.

🏨 Analogy: the hotel key card A delivery company (the external system) has a standing contract with a hotel (the API provider). The driver doesn't check in like a guest — he shows his company badge (Client ID) and signed contract code (Client Secret) at the front desk (token endpoint). The desk issues a key card (access token) that opens only the doors the contract allows and expires after a while. Expired card? Back to the desk for a new one. Nobody ever hands over a master key (a password).

The two steps, always

  1. Get a token: POST your Client ID + Secret to the token endpoint → receive an access token.
  2. Use the token: attach it to every API call as Authorization: Bearer <token>.

Three properties to memorize

  1. No refresh token, by design. Need a new token? Present the credentials again. (The diagram's last note.)
  2. The token acts as a fixed identity. Inbound to Salesforce: the Run As user. At Auth0: the M2M application itself, capped by its scopes.
  3. Whoever holds ID + Secret is the app. There's no second factor — which drives every security rule in this phase.
🔒 The one fact to never forget Anyone with the Client ID + Secret can mint tokens and act with the app's full access. That's why we: (1) grant minimal permissions, (2) store secrets only in encrypted stores (External Credential principals, secrets managers), (3) rotate periodically — and immediately upon any suspected leak.
🧠 Check yourself: why does this flow have no refresh token, while the web server flow does?
A refresh token exists to avoid bothering a human to log in again. Here there is no human — the app can simply present its credentials again, so a refresh token would add risk with zero convenience.

B4 — The Apex service class

Developer Console → File → New → Apex Class → Auth0UserService. Full source is in resources/module-2-client-credentials/apex/Auth0UserService.cls; the heart of it:

// One reusable method builds every request — CRUD methods just fill in the verb.
public with sharing class Auth0UserService {

    private static final String USERS_PATH = 'callout:Auth0_Mgmt_API/api/v2/users';

    public class Auth0Exception extends Exception {}

    // C — returns the new user's id (looks like "auth0|68a1…")
    public static String createUser(String email, String password) {
        Map<String, Object> body = new Map<String, Object>{
            'email'      => email,
            'password'   => password,
            'connection' => 'Username-Password-Authentication' };
        HttpResponse res = send('POST', USERS_PATH, JSON.serialize(body));
        Map<String, Object> user =
            (Map<String, Object>) JSON.deserializeUntyped(res.getBody());
        return (String) user.get('user_id');
    }

    // R
    public static Map<String, Object> getUser(String userId) {
        HttpResponse res = send('GET', userPath(userId), null);
        return (Map<String, Object>) JSON.deserializeUntyped(res.getBody());
    }

    // U — stores extra info in user_metadata, Auth0's "custom fields"
    public static void updateUserPlan(String userId, String plan) {
        String body = JSON.serialize(new Map<String, Object>{
            'user_metadata' => new Map<String, Object>{ 'plan' => plan } });
        send('PATCH', userPath(userId), body);
    }

    // D — Auth0 answers 204 No Content on success
    public static void deleteUser(String userId) {
        send('DELETE', userPath(userId), null);
    }

    private static String userPath(String userId) {
        return USERS_PATH + '/' + EncodingUtil.urlEncode(userId, 'UTF-8');
    }

    // The single door every request goes through: build, send, fail loudly.
    private static HttpResponse send(String method, String endpoint, String jsonBody) {
        HttpRequest req = new HttpRequest();
        req.setEndpoint(endpoint);
        req.setMethod(method);
        if (jsonBody != null) {
            req.setHeader('Content-Type', 'application/json');
            req.setBody(jsonBody);
        }
        HttpResponse res = new Http().send(req);
        if (res.getStatusCode() >= 300) {
            throw new Auth0Exception(method + ' ' + endpoint + ' failed: '
                + res.getStatusCode() + ' ' + res.getBody());
        }
        return res;
    }
}
💡 Naming conventions at work The class is named for what it serves (Auth0UserService), methods for what they do (createUser), and one private send() holds all plumbing so the public methods read like plain English. Copy this shape for every integration you ever write.

C3 — Why we need "threading": triggers can't call out
💡 The rule and the reason A trigger runs inside the database save. If it paused to wait on an external API (which might take seconds… or hang), it would hold the record — and parts of the database — locked. So Salesforce forbids synchronous callouts from triggers. The pattern: the trigger only enqueues a background job (Queueable Apex — this is the "threading"), and the job makes the callout seconds later, writing results back to the record.

Sequence: User ticks checkbox → trigger fires → enqueues job → save completes instantly → moments later the job calls Auth0 → job updates the Contact with the id/status.

C4 — The Queueable + the trigger

Full source in resources/module-2-client-credentials/apex/. The Queueable (note Database.AllowsCallouts — the permission slip for callouts in async jobs):

public with sharing class Auth0ProvisioningQueueable
        implements Queueable, Database.AllowsCallouts {

    private List<Id> contactIds;
    public Auth0ProvisioningQueueable(List<Id> contactIds) { this.contactIds = contactIds; }

    public void execute(QueueableContext ctx) {
        List<Contact> toUpdate = new List<Contact>();
        for (Contact c : [SELECT Id, Email FROM Contact
                          WHERE Id IN :contactIds AND Email != null]) {
            try {
                String tempPassword = 'Aa1!' + EncodingUtil.convertToHex(
                    Crypto.generateAesKey(128)).left(12);      // random starter password
                String userId = Auth0UserService.createUser(c.Email, tempPassword);
                toUpdate.add(new Contact(Id = c.Id,
                    Auth0_User_Id__c = userId,
                    Provisioning_Status__c = 'Provisioned ' + Date.today().format()));
            } catch (Exception e) {
                toUpdate.add(new Contact(Id = c.Id,
                    Provisioning_Status__c = ('ERROR: ' + e.getMessage()).left(255)));
            }
        }
        update toUpdate;
    }
}

And the trigger — tiny on purpose (triggers should decide when, classes decide what):

trigger ContactTrigger on Contact (after insert, after update) {
    List<Id> needProvisioning = new List<Id>();
    for (Contact c : Trigger.new) {
        Contact old = Trigger.isUpdate ? Trigger.oldMap.get(c.Id) : null;
        Boolean justTicked = c.Provision_Portal_Access__c
            && (old == null || !old.Provision_Portal_Access__c);
        if (justTicked && c.Auth0_User_Id__c == null) {
            needProvisioning.add(c.Id);
        }
    }
    if (!needProvisioning.isEmpty()) {
        System.enqueueJob(new Auth0ProvisioningQueueable(needProvisioning));
    }
}
⚠️ Error handling is not optional The try/catch writes failures onto the record (Provisioning_Status__c) instead of exploding invisibly at 2 AM. In production you'd also log to a custom object — the habit starts here.

The Web Server Flow, Explained

Module 2 answered "how does a machine get in?" Module 3 answers a different question: how does an app act as a specific person — with their permissions, their data, their audit trail — without ever learning their password?

🎟️ Analogy: the coat check You (the user) hand your coat (your identity) directly to the coat-check attendant (the platform's own login page) — never to the friend (the app) who'll pick it up for you. The attendant gives a claim ticket (authorization code). Your friend presents the ticket plus their own ID (Client ID + Secret) to collect the coat (access token). PKCE is the ticket torn in half: your friend left one half (the code_challenge) when ordering; only the person holding the matching half (the code_verifier) can redeem it — a pickpocket who steals the ticket gets nothing. And the refresh token is a standing arrangement: "you may pick up for this guest again without asking each time."

The moving parts

  1. Send the user to log in at the platform's authorize endpoint (with your client_id, redirect_uri, scopes, and the PKCE code_challenge).
  2. The user logs in on the platform's page and clicks Allow on the consent screen.
  3. The browser is redirected to your callback URL carrying a single-use, short-lived authorization code.
  4. Your app exchanges the code (+ its secret + the PKCE code_verifier) at the token endpoint → access token + refresh token.
  5. When the access token expires, the refresh token flow gets a new one silently — no human needed.
Diagram source (Mermaid) — also in resources/shared/diagrams/web-server-flow.mmd 
sequenceDiagram
    App->>AZ: GET /services/oauth2/authorize (response_type=code, client_id, redirect_uri, code_challenge)
    AZ->>User: login page + consent screen
    User->>AZ: credentials (+MFA), click Allow
    AZ-->>App: redirect to callback ?code=AUTH_CODE (single-use!)
    App->>T: POST /services/oauth2/token (grant_type=authorization_code, code, client_id, client_secret, redirect_uri, code_verifier)
    T-->>App: access_token + refresh_token
    App->>T: later: grant_type=refresh_token → fresh access_token, no login

Why the extra "code" step? (everyone asks)

The browser is the least trustworthy link — extensions, history, and redirects all see its traffic. So the flow splits: the front channel (browser) only ever carries the short-lived, single-use code, while real tokens travel on the back channel (app server → token endpoint, protected by the client secret and the PKCE verifier). Steal the front channel and you hold a voucher you can't redeem.

🔒 Refresh tokens are crown jewels Long-lived + silently usable = maximum damage if stolen. Store them encrypted (Salesforce does this for you in External Credentials), enable rotation where supported, and know the revoke levers (topic 9).
🧠 Check yourself: why does this flow issue a refresh token when Module 2's flow refused to?
A refresh token exists to avoid bothering a human to log in again. Module 2 had no human — the app just re-sends its credentials. Here a human logged in once, and the refresh token lets the app keep working without dragging them back to the login screen.

/** Uploads plain text as a file. Returns the path Dropbox stored it under. */
public static String uploadTextFile(String path, String content) {
    HttpRequest req = new HttpRequest();
    req.setEndpoint('callout:Dropbox_Content/2/files/upload');   // content host!
    req.setMethod('POST');
    // Parameters go in a HEADER because the body carries the file bytes.
    req.setHeader('Dropbox-API-Arg', JSON.serialize(new Map<String, Object>{
        'path' => path, 'mode' => 'add', 'autorename' => true }));
    req.setHeader('Content-Type', 'application/octet-stream');
    req.setBody(content);                                        // the raw file
    HttpResponse res = new Http().send(req);
    if (res.getStatusCode() >= 300) {
        throw new DropboxException('Upload failed: '
            + res.getStatusCode() + ' ' + res.getBody());
    }
    Map<String, Object> meta =
        (Map<String, Object>) JSON.deserializeUntyped(res.getBody());
    return (String) meta.get('path_display');
}

Module 4 · JWT Bearer

The JWT Bearer Flow, Explained

Server-to-server like Module 2 — but with one structural upgrade: no secret ever crosses the network. Instead of presenting a password-like secret, the app signs a short message with a private key, and the platform verifies the signature with a registered certificate.

💌 Analogy: the sealed letter A courier arrives at the castle with a letter: "Admit my messenger to act for the Duke. Valid until sundown." It's closed with the Duke's wax seal (the signature, made with his signet ring — the private key). The guards hold an impression of that seal (the public certificate, registered in advance) — enough to verify any seal, never enough to make one. A forged letter fails the seal check; an old letter fails "valid until sundown" (exp); a letter addressed to another castle fails the address check (aud). And the Duke never told anyone a password.

What the signed message says (the claims)

// A JWT payload is just a tiny JSON of "claims":
{
  "iss": "who is speaking",      ← the app (consumer key / service account email)
  "sub": "acting as whom",       ← a username (inbound to Salesforce)
  "aud": "for which server",      ← so it can't be replayed elsewhere
  "exp": "until when"             ← minutes, not months — stolen ≈ already dead
}

The whole thing is signed with RSA SHA-256 (RS256) and POSTed to the token endpoint with grant_type=urn:ietf:params:oauth:grant-type:jwt-bearer and the JWT in a parameter literally named assertion.

Diagram source (Mermaid) — also in resources/shared/diagrams/jwt-bearer-flow.mmd 
sequenceDiagram
    S->>K: Sign the claim set (RSA SHA-256)
    K-->>S: signed JWT — a tamper-proof "sealed letter"
    S->>T: POST /services/oauth2/token (grant_type=…jwt-bearer, assertion=JWT)
    T->>T: verify signature with the registered X.509 certificate + check pre-authorization
    T-->>S: access_token — NO refresh token; sign a fresh JWT next time
    S->>API: Authorization: Bearer … → JSON

Three properties to memorize

  1. No login screen, ever — so consent must exist in advance: pre-authorization (inbound to Salesforce) or resource sharing (Google — you'll share a calendar with the app).
  2. No refresh token. Need a new access token? Sign a fresh JWT — that is the refresh.
  3. The private key never travels. Salesforce/Google store only the public certificate — useless to a thief. This is why security teams and CI/CD pipelines standardize on this flow.
🧠 Check yourself: Module 2's flow also had no human. Why would a security team force this flow instead?
Module 2's Client Secret is a shared secret: it sits in two places and crosses the network on every token request — every copy is a leak opportunity. Here only a signature travels; the private key stays in one vault, and the server holds a certificate that can verify but never forge. Fewer copies, shorter lifetimes, no secret in transit.

Inbound: Salesforce as the Server

Three building blocks: a key pair you generate, an External Client App that trusts its certificate, and the pre-authorization policy. Commands are also in resources/module-4-jwt-bearer/scripts/jwt-inbound-commands.sh.

Step 1 — Generate the certificate pair (3 terminal commands)

In a terminal (macOS/Linux have openssl built in; on Windows it ships with Git for Windows), inside an empty folder like ~/jwt-lab:

# 1. The PRIVATE key — the signet ring. This file is the crown jewel.
openssl genrsa -out server.key 2048

# 2. A certificate request (asks name/country questions — Enter through them for a lab)
openssl req -new -key server.key -out server.csr

# 3. Self-sign into an X.509 certificate valid 1 year — the public "wax impression"
openssl x509 -req -sha256 -days 365 -in server.csr -signkey server.key -out server.crt

Result: server.key 🔑 private — never leaves this machine and server.crt public — safe to upload.

🔒 Handling rule, upgraded In Module 2 the secret was a password; here server.key is the only signet ring. Never email it, never commit it, never upload it anywhere. Production home: a secrets vault or the CI system's protected storage. Suspected leak → remove the certificate from the app and generate a new pair.
Step 2 — Configure the ECA: enable JWT + upload the certificate
  1. Reuse the training app (or create one per the Module 1 deep-dive). Edit Settings → OAuth Settings: scopes Manage user data via APIs (api) + Perform requests at any time (refresh_token, offline_access) — odd for a no-refresh-token flow, but the Salesforce CLI keeps its sessions alive under that scope, and CLI guides include it.
  2. In Flow Enablement, check Enable JWT Bearer Flow — the Module 1 checkbox whose time has come.
  3. Use the Upload Files control that appears with it and upload server.crt. Only the .crt — if you're ever about to upload a .key, stop.
  4. Save; note the Consumer Key (the secret exists but this flow never uses it).
⚠️ Certificate size limit The certificate can't exceed 4 KB (official limit). Ours won't; oversized real-world certs usually fit as DER-encoded files.
Step 3 — Pre-authorize the user (the step everyone forgets)

No consent screen exists in this flow, so approval must exist in advance — the docs call it "prior approval of the app". You already rehearsed this in Module 3's homework:

  1. App → Policies tab → Permitted Users = Admin approved users are pre-authorized. Save.
  2. Grant the JWT's sub user access via a permission set or profile (add this ECA to its app access). The Module 2 integration user is perfect — or your own user for the lab.
💡 Why this matters Forgetting it produces the most-Googled error in Salesforce DevOps: invalid_grant: user hasn't approved this consumer. After today you'll fix it in one minute, forever.

    // Paste yours — the only configuration this class needs:
    private static final String SERVICE_ACCOUNT_EMAIL =
        'sf-kickoff-scheduler@YOUR-PROJECT.iam.gserviceaccount.com';
    private static final String CALENDAR_ID = 'you@gmail.com';

    private static final String CERT_NAME  = 'GoogleSA';
    private static final String TOKEN_URL  = 'https://oauth2.googleapis.com/token';
    private static final String SCOPE      = 'https://www.googleapis.com/auth/calendar.events';
    private static final String EVENTS_URL = 'https://www.googleapis.com/calendar/v3/calendars/';

    // STEP 1+2 of the diagram: write claims, sign with the org's certificate,
    // exchange for a Bearer token. This is the whole flow, in four statements.
    private static String getAccessToken() {
        Auth.JWT jwt = new Auth.JWT();
        jwt.setIss(SERVICE_ACCOUNT_EMAIL);
        jwt.setAud(TOKEN_URL);
        jwt.setAdditionalClaims(new Map<String, Object>{ 'scope' => SCOPE });
        Auth.JWS jws = new Auth.JWS(jwt, CERT_NAME);
        return new Auth.JWTBearerTokenExchange(TOKEN_URL, jws).getAccessToken();
    }

    // C — returns the new event's id
    public static String createEvent(String title, Datetime startAt, Datetime endAt) {
        Map<String, Object> body = new Map<String, Object>{
            'summary' => title,
            'start' => new Map<String, Object>{ 'dateTime' => rfc3339(startAt) },
            'end'   => new Map<String, Object>{ 'dateTime' => rfc3339(endAt) } };
        HttpResponse res = send('POST', eventsPath(null), JSON.serialize(body));
        return (String) ((Map<String, Object>)
            JSON.deserializeUntyped(res.getBody())).get('id');
    }

    // R
    public static Map<String, Object> getEvent(String eventId) {
        HttpResponse res = send('GET', eventsPath(eventId), null);
        return (Map<String, Object>) JSON.deserializeUntyped(res.getBody());
    }
    // …updateEventTitle / deleteEvent in Session C; full source in resources.
}

Modules 2–4 taught how systems prove who they are. But every project's Salesforce side eventually needed a developer's tool: Apex. This module removes even that. Flow — the tool you already live in as an admin — can call APIs, receive calls, and publish events, with zero code.

📦 Analogy: ways to move a parcel Think of each integration method as a delivery option. HTTP Callout is a courier with signature-required: you send the request and wait for the answer before the flow continues. Platform Events and Outbound Messages are dropping a letter in the mailbox: fire-and-forget — the flow moves on, delivery happens on its own. And the inbound REST Actions API is the customer walking into your store: an outside system rings the bell and your flow serves them.

Inbound: Your Flow Becomes an API

Here's the quiet superpower buried in every org: any active autolaunched flow is already a REST endpoint. No Apex REST class, no middleware — an external system POSTs to the Actions API, your flow runs, and its output variables come back as the response.

The recipe (from Salesforce's own docs)

POST {{my_domain}}/services/data/v{{api_version}}/actions/custom/flow/Your_Flow_API_Name
Authorization: Bearer <access_token>     ← any token from Modules 2–4!
Content-Type: application/json

{ "inputs": [ { "YourInputVariable": "value" } ] }

Three requirements: the flow is autolaunched (no screens — nobody's there to click), it's Active, and its variables are explicitly marked Available for input / Available for output. That checkbox is the API contract.

Build it — "Lead Intake API", every click

The business story: a partner website wants to push leads into your org. Instead of exposing objects directly, you expose one controlled flow that applies your rules. Three elements, ~10 minutes.

Step 1 — Create the flow shell
  1. Setup → Quick Find: Flows → New Flow.
  2. Choose Start From Scratch → Next → flow type: Autolaunched Flow (No Trigger) → Create.
  3. You land on an empty canvas with a Start and End element. Everything you add goes between them.
Step 2 — Create the three variables (the API contract)

Open the Toolbox (the panel on the left — if you see tabs, pick Manager) and click New Resource. Repeat three times with these values:

Resource TypeAPI NameData TypeAvailability checkboxes
VariableCompanyNameText✅ Available for input
VariableContactLastNameText✅ Available for input
VariableNewLeadIdText✅ Available for output
💡 Those two checkboxes ARE the API "Available for input" makes a variable a valid key in the request's inputs JSON; "Available for output" puts it in the response's outputValues. Spelling and capitalization of the API names must match the JSON exactly — they're contract terms.
Step 3 — One Create Records element
  1. On the canvas, click the ＋ between Start and End → choose Create Records.
  2. Label: Create the Lead (API name auto-fills).
  3. How Many Records to Create: One · How to Set the Record Fields: Use separate resources, and literal values.
  4. Object: Lead. Then add three field rows:
    • LastName ← {!ContactLastName}
    • Company ← {!CompanyName}
    • LeadSource ← Web (typed as a literal)
  5. Near the bottom, check Manually assign variables — a Store Lead ID box appears. Set it to {!NewLeadId}. That's how the created record's ID reaches your output variable.
  6. Click Done.
Step 4 — Save, test inside Flow Builder, activate
  1. Save → Flow Label: Lead Intake API. Note the auto-generated Flow API Name Lead_Intake_API — that exact string is the URL segment Postman will call.
  2. Before touching Postman, click Debug → the debug panel asks for your input variables (because they're marked available for input). Enter a test company and last name → Run. The debug detail should show the Create Records element succeeding and NewLeadId being set. Test in the builder first, then over the API — two layers of certainty.
  3. Click Activate. Only Active flows are invocable from REST.
✅ The gate you control The partner can't touch anything except what this flow does. Want dedupe rules, a default owner, a "partner" record type? Add elements — the API contract (inputs/outputs) stays identical. That's why teams expose flows instead of raw objects.

  "actionName": "Lead_Intake_API",
  "isSuccess": true,
  "outputValues": { "NewLeadId": "00Q…" }
} ]

Now open Leads in your org: there's Weber / Acme Partner GmbH, Created By = your integration user — Module 2's audit-trail lesson, alive and well. One request created a record through your business logic.

💡 Batch for free The inputs array takes multiple objects — each one runs the flow once. Three leads in one call, no extra work. (That's also your first hint about limits: 25,000 leads do not belong in this pattern — see the honest-limits callout in topic 4.)

Break it on purpose 🧪

  • Deactivate the flow and re-run → the call fails; only Active flows are invocable. Reactivate.
  • Typo an input ("Company" instead of "CompanyName") → the error names the unknown variable. Input names are case-sensitive contract terms.
  • Remove Create on Lead from the integration user's permission set, wait a minute, re-run → isSuccess: false with an access error in the response. Restore it. Same lever, fourth module in a row.

🏗 Project Session A — Amadeus Setup + the Credential Chain

The project: a "Travel Desk" — reps visit clients, and operations wants each Account stamped with its destination city's IATA code and coordinates (the codes travel agents and booking tools actually use). The data comes live from Amadeus — the travel-technology company whose APIs power real airlines and agencies, with a free self-service test tier. Session A: register the app, prove the API in Postman, and build the real OAuth credential chain in Salesforce.

✅ Session A exit criteria A token and a city search both return 200 in Postman, and the Amadeus credential chain (External Credential → Principal → permission set → Named Credential) is saved in Salesforce. ~45 minutes.
A1 — Register the app (the universal ritual, fifth platform)
  1. Create a free account at developers.amadeus.com → My Self-Service Workspace → Create New App, e.g. Salesforce Travel Desk.
  2. Copy the API Key (= Client ID) and API Secret (= Client Secret). 🔑 secret! Same handling rules as every module: never in chat, docs, or code.
💡 Say the pattern out loud Register app → receive Client ID + Secret → exchange for tokens. Salesforce, Auth0, Dropbox, Google — and now Amadeus. Five platforms, one ritual. You could do this in your sleep now, which is exactly the point.
A2 — Prove it in Postman (token → city search → save the JSON)

Request 1 — the token. Build it like Module 2, Request 1 — POST, Body → x-www-form-urlencoded, three rows — with amadeus_client_id / amadeus_client_secret set in the Variables tab. The credentials ride in the body (note that for A3's "credentials in request body" toggle). Compare with Module 2 line by line — only the company changed:

POST https://test.api.amadeus.com/v1/security/oauth2/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
client_id={{amadeus_client_id}}
client_secret={{amadeus_client_secret}}

Response: an access_token with expires_in around 1799 — Amadeus tokens live ~30 minutes. (No refresh token, as you'd predict for this flow — re-request when it dies. In Salesforce, the Named Credential engine will do that for you.)

Request 2 — City Search (the API our flow will call) — a GET with Authorization → Bearer {{amadeus_access_token}}; the keyword and max query params show as editable rows in the Params tab:

GET https://test.api.amadeus.com/v1/reference-data/locations/cities?keyword=PARIS&max=3
Authorization: Bearer {{amadeus_access_token}}

The response is an array — richer than anything the earlier modules parsed:

{
  "data": [
    {
      "type": "location", "subType": "city",
      "name": "PARIS", "iataCode": "PAR",
      "geoCode": { "latitude": 48.85341, "longitude": 2.3488 }
    }
  ], …more matches…
}

Save your real response JSON. In Postman's response pane, click the Copy icon at the top-right of the body — or use Save Response → Save to a file — and keep it. Session B pastes it into Flow Builder's "Provide Sample Response" step, and the keys (data, name, iataCode, geoCode.latitude) become your typed Flow variables. Clean camelCase keys: Flow-safe.

⚠️ Leave include=AIRPORTS out — on purpose City Search has an optional include=AIRPORTS parameter that returns airports keyed by their codes — "CDG": {…}, "ORY": {…}. Those are dynamic JSON keys: a sample from Paris builds a structure that Madrid's response won't match. It's the exact trap from the troubleshooting table, sitting in a real API — the homework sends you to meet it deliberately.
A3 — The credential chain in Salesforce (real OAuth, zero code)

Module 2's pattern, minus the audience quirk — feel how fast it goes the second time:

  1. External Credential: Label/Name Amadeus · Authentication Protocol OAuth 2.0 · Authentication Flow Type Client Credentials with Client Secret · Identity Provider URL https://test.api.amadeus.com/v1/security/oauth2/token · Scope blank. If your release shows a "pass client credentials in request body"-style option, turn it on — Amadeus expects the credentials as form-body parameters (Postman just proved that works).
  2. Principal AmadeusApp: paste the API Key and API Secret. Encrypted on save, as always.
  3. Permission set (reuse Flow Integrations Access or Module 2's) → External Credential Principal Access → add Amadeus – AmadeusApp → assigned to you.
  4. Named Credential: Label/Name Amadeus · URL https://test.api.amadeus.com · External Credential Amadeus · Generate Authorization Header ✅ · Enabled for Callouts ✅.
💡 What the engine now does for your flow — invisibly Every callout: check for a cached Amadeus token → if missing or past its ~30 minutes, POST the client-credentials request → attach Authorization: Bearer … → send. Token caching, renewal, and header injection: the entire "API calling procedure" chapter of a developer book, handled by configuration you built in four steps.
✅ Variation: APIs with no auth at all Some public APIs (e.g. the Frankfurter exchange-rate API) need no credentials — but HTTP Callout still requires a Named Credential. The pattern: External Credential with the Custom protocol and an empty principal (it still gates which users' flows may call out, and keeps every external URL visible in Setup — governance even without secrets). Homework #2 has you build one.

🏗 Project Session B — The Travel Desk Flow

The full arc in one build: authenticate → call → parse → write back → handle failure. A rep opens an Account, types the client's city, and the flow fetches the official IATA city code and coordinates from Amadeus — through real OAuth it never has to think about — and stamps them on the record. Every click is below; build it exactly once with the guide, then once without.

✅ Session B exit criteria From an Account, the flow finds a city live in Amadeus, shows it for confirmation, and updates the Account's travel fields — with a working fault screen when things go wrong. ~60 minutes.
B1 — Two fields on Account (admin work)
  • Travel_City_Code__c — Text(3). The IATA city code booking tools use (PAR, NYC, DXB).
  • Travel_City_Name__c — Text(80). The matched city name, so humans can sanity-check the code.

Add both to the Account page layout.

B2 — Flow shell, the recordId variable, and Screen 1
  1. Setup → Flows → New Flow → Start From Scratch → Screen Flow → Create.
  2. Toolbox → New Resource: Resource Type Variable · API Name exactly recordId · Data Type Text · ✅ Available for input → Done. That magic name is how Lightning record pages hand the flow the Account it's sitting on — the same input-variable contract as topic 2's API, used by the page instead of Postman.
  3. Click ＋ after Start → Screen. Label: Ask City.
  4. From the component list on the left, drag a Text component onto the screen canvas. Configure it: Label City keyword · API Name CityKeyword · ✅ Require · Help Text: "Start of the city name, 2–10 characters — e.g. PAR or PARIS." (Those bounds are from Amadeus's own spec for the keyword parameter.) → Done.
B3 — Create the HTTP Callout action (the wizard, click by click)
  1. Click ＋ after the screen → Action. In the action dialog, choose Create HTTP Callout (a button/option alongside the action search).
  2. Wizard part 1 — the service: Name Amadeus_Service · Named Credential: Amadeus (Session A's real-OAuth credential — if it's missing from this picklist, your user lacks principal access; see troubleshooting) → Next.
  3. Wizard part 2 — the invocable action:
    • Label Search Cities (API name auto-fills Search_Cities) · Method GET · URL Path /v1/reference-data/locations/cities — path only, no ?.

  5. You're returned to the action's configuration dialog (this is the instance of your new action in this flow). Set: keyword = {!CityKeyword} · max = 1 (switch the optional parameter on to set it) → Done.
💡 What just happened Flow generated an External Services registration + typed Apex classes from one pasted JSON (check Setup → External Services → Amadeus_Service). Its response is now a first-class Flow resource: {!Search_Cities.2XX…}. With max=1, taking the top match keeps v1 honest — autocomplete APIs rank best-first; homework upgrades to a pick-list of 5.
B4 — Parse the array: the Loop is your JSON parser
  1. Toolbox → New Resource four times: CityName (Text), CityCode (Text), CityLat (Number), CityLon (Number) — plain variables, no availability boxes needed.
  2. Click ＋ after the action → Loop. Label: Read Matches. Collection Variable: drill into Outputs from Search_Cities → 2XX → data. Direction: first to last → Done. The canvas now shows two branches: For Each (runs per item) and After Last (runs when done).
  3. On the For Each branch, click ＋ → Assignment. Label: Copy City Fields. Add four rows — the loop's current item is a resource named after the loop ({!Read_Matches}), so:
    • {!CityName} Equals {!Read_Matches.name}
    • {!CityCode} Equals {!Read_Matches.iataCode}
    • {!CityLat} Equals {!Read_Matches.geoCode.latitude}
    • {!CityLon} Equals {!Read_Matches.geoCode.longitude}

With max=1 the loop runs exactly once — training wheels on. The mental model to keep: in Flow, a Loop + Assignment is what for(…) + field access is in Apex.

B5 — Confirm, write back, and handle failure (on the After Last branch)
  1. On the loop's After Last branch, click ＋ → Screen. Label Confirm; drag a Display Text component in and paste (insert the variables with the {!…} resource picker): 
    Found: {!CityName} ({!CityCode})
Coordinates: {!CityLat}, {!CityLon}

Save this to the Account?
  2. ＋ after Confirm → Update Records. Label Stamp the Account · How to Find Records: Specify conditions to identify records, and set fields individually · Object Account · Condition: Id Equals {!recordId} · Set Field Values: Travel_City_Code__c ← {!CityCode}, Travel_City_Name__c ← {!CityName} → Done.
  3. Fault paths, both of them: open the Search Cities element's menu (the ⌄/⋯ on the element) → Add Fault Path → on the fault branch add a Screen with a Display Text: Something went wrong: {!$Flow.FaultMessage}. Repeat for Stamp the Account (its fault screen can reuse the same message). APIs time out, tokens get revoked, records get locked — a Travel Desk that fails silently books nobody anywhere.
  4. Save as Travel Desk → Debug → enter PAR (leave recordId blank in debug, or paste any Account Id to test the update too) → watch the debug log: the callout, the parsed values, the update. Then Activate.
  5. Make it real: open any Account → gear icon → Edit Page (Lightning App Builder) → drag the Flow component onto the page → select Travel Desk → check Pass record ID into this variable (under recordId) → Save → Activate the page if prompted.

The payoff: open any Account → type PAR → "Found: PARIS (PAR)" → Next → the fields are stamped. Behind that click: a cached OAuth token, a live API in Amadeus's test cloud, typed JSON parsing, and a DML write — and the person who built all of it never wrote a line of code. 🎉

Troubleshooting Guide

Flow hides the plumbing, so its errors point at configuration — usually credentials, paths, or the sample JSON.

HTTP Callout (outbound)

SymptomCause → fix
Callout action won't run in a record-triggered flow / an error about callouts in the transactionIt's not on the Run Asynchronously path. Callouts can't share the save's transaction — Module 2's Queueable rule, Flow edition. Rebuild the callout steps on the async path.
Named Credential missing from the HTTP Callout picklistYour user lacks External Credential Principal Access for it — the same permission-set lever as every module. (Also confirm Enabled for Callouts on the Named Credential.)
Sample JSON rejected, or the data structure is missing fieldsKeys with spaces or special characters can't become variable names — pick a cleaner API or wrap it in Apex; or your sample was too minimal — paste a real, full response from Postman, never a hand-typed sketch.
A value comes back empty at runtime that the sample clearly hadDynamic keys: the runtime response's keys don't match the sampled ones — e.g. Amadeus's include=AIRPORTS returns airports keyed by their codes ("CDG": {…}), so a Paris sample can't parse a Madrid response. Restructure the call to avoid dynamic keys, or hand the parsing to Invocable Apex.
Auth-style error calling a public APIUncheck Generate Authorization Header on the no-auth Named Credential — there's no token to generate, so don't ask for one.
401/403 calling Auth0 from the flowModule 2's table applies unchanged: principal credentials, scopes on the M2M app, the audience body parameter with its trailing slash. The flow changed; the OAuth didn't.
invalid_client from Amadeus's token endpointRe-copy the API Key/Secret onto the principal — and check the External Credential's pass client credentials in request body option is on: Amadeus expects the credentials as form-body parameters, exactly as your working Postman request sent them.
Works in Debug, fails for real usersReal users need the permission set granting principal access too — Debug ran as you.

Inbound (REST → Flow)

SymptomCause → fix
404 on /actions/custom/flow/…Wrong flow API name (it's the underscored auto-generated one, not the label) — or the flow isn't Active.
Error naming one of your inputsThe variable isn't marked Available for input, or the JSON key's spelling/case doesn't match exactly. The checkbox is the contract.
isSuccess: false with an access errorThe token's user (Module 2's integration user) lacks object/field permissions for what the flow does — fix the permission set, not the flow.
